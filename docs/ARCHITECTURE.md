# ARCHITECTURE.md: Direct Buy

## 1. Технологический стек
Проект строится как единая кодовая база вокруг экосистемы **TypeScript / Node.js**:
- **Язык:** TypeScript (Strict mode).
- **Runtime:** Node.js 22 LTS. Локальный baseline зафиксирован в `.node-version`/`.nvmrc`, а минимальный поддерживаемый диапазон объявлен в `package.json`.
- **Пакетный менеджер:** `pnpm` через Corepack. Версия закреплена в `package.json` (`packageManager`), lockfile проекта - только `pnpm-lock.yaml`.
- **Telegram Bot:** `grammY` — мощный и быстрый фреймворк для написания ботов на TS/JS. Поддерживает удобную работу с сессиями и Middleware.
- **Admin Panel / Web Backend:** `Next.js 16+` (App Router) + React 19 (Current LTS).
- **UI Библиотека админки:** TailwindCSS + `shadcn/ui` (или аналоги).
- **База данных:** PostgreSQL.
- **ORM:** `Prisma` или `Drizzle ORM` (для строгой типизации запросов из Next.js и Бота).
- **Брокер сообщений/Кеширование:** `Redis`.

## 2. Оркестрация и SLA (Background Jobs)
Сложная логика SLA (таймеры на 2, 3 и 12 часов) не может полагаться на `setTimeout` в Node.js из-за риска падения сервера.
- **Выбранное решение:** `BullMQ` поверх `Redis`.
- **Механика работы:** 
  - При выставлении объекта на "Сбор предложений" (Аукцион), в систему ставится `Delayed Job` на 12 часов (событие `CLOSE_AUCTION`).
  - Если клиент собирает 5 предложений быстрее, задача отменяется (removed) через её ID.
  - Если клиент переходит на этап загрузки документов, ставится `Delayed Job` на 2 часа (`SLA_DOCS_UPLOAD`). Если клиент не успел, воркер обрабатывает событие и кидает Alert Менеджеру в отдельный чат.

## 3. Хранение данных (Media Storage)
Документы (Паспорта, ЕГРН) и Фотографии (7-10 штук) являются чувствительными данными.
- **Текущее решение:** Сохранение на **Локальный диск (Docker Volume)** виртуального сервера (VPS).
- **Безопасность:**
  - Файлы не публикуются в открытый веб.
  - Доступ к файлам для Инвестора осуществляется через защищенные API-эндпоинты (Next.js), которые проверяют авторизацию и то, что именно этот Инвестор выиграл данный лот.
- **Масштабирование в будущем:** С ростом проекта или при переходе в облако (K8s) хранилище будет заменено на S3-совместимое API (Timeweb Cloud S3, Yandex Object Storage).

## 4. Развертывание (Deployment)
- **Инфраструктура:** Собственный VPS-сервер.
- **Оркестрация:** Контейнеризация всех компонентов через `Docker` и управление через `docker-compose.yml`.
- **Структура контейнеров:**
  - `bot` (grammY процесс, слушает Telegram или работает через Webhooks).
  - `admin-panel` (Next.js сервер).
  - `worker` (Скрипт-подписчик BullMQ на процессинг отложенных задач).
  - `postgres` (БД).
  - `redis` (Очереди, кеш, сессии).
- **Reverse Proxy:** `Caddy` или `Nginx` для распределения трафика, выдачи SSL-сертификатов (Let's Encrypt) и терминации HTTPS.

## 5. Структура БД (Core Entities Overview)
- **`User`**: Хранит telegram_id, роль (Овнер, Брокер, Инвестор, Менеджер), профиль (ФИО, телефон).
- **`InvestorProfile`**: Параметры поиска (Мин/Макс бюджет, Районы, Тип объектов), флаг верификации (`isVerified: boolean`).
- **`Lot` (Объект)**: 
  - Данные: Адрес, Метраж, Этаж, Комнатность, ЮрСтатус.
  - Финансы: Скрытая "Ожидаемая сумма".
  - Статус: (Draft, Auction, WaitingChoice, WaitingDeposit, WaitingDocs, DocsAudit, ReadyToDeal, Canceled, Sold).
- **`Bid` (Ставка Инвестора)**: Привязка Инвестора к Лоту. Содержит предложенную сумму `Total`.
- **`Media`**: Ссылки на физические файлы на сервере, привязанные к Лоту по типам (`PHOTO`, `PASSPORT`, `EGRN`, и тд).
